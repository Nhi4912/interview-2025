# React/TypeScript Coding Standards

> **Scope**: ALL React/TypeScript code generated by Claude — doc examples in `docs/fe-track/`, standalone projects, code reviews, refactoring.
> These rules are **MANDATORY**, not suggestions. Verify against Section 9 BANNED checklist before finalizing any code.

---

## Section 1: TypeScript Strict Rules (always apply)

1. **`const` only** — `const` for all variable declarations.
   BANNED: `let` (refactor to derived values, `.map()`, `.reduce()`, `.filter()`, or separate `const`).
   BANNED: `var` (always).
   EXCEPTION: `let` inside `for...of` / `for` loops ONLY when `const` is syntactically impossible.

   ```typescript
   // ✅ Correct
   const items = data.filter((item) => item.active)
   const total = prices.reduce((sum, price) => sum + price, 0)

   // ❌ BANNED
   let filtered = []
   for (let i = 0; i < data.length; i++) { filtered.push(data[i]) }
   ```

2. **No `any`** — every value must have a concrete type.
   BANNED: `any` in all positions (parameter, return, generic, type assertion).
   Use proper generics, union types, or `unknown` with immediate type narrowing.

   ```typescript
   // ✅ Correct
   function parse<T>(input: string, schema: z.ZodSchema<T>): T { ... }

   // ❌ BANNED
   function parse(input: any): any { ... }
   ```

3. **No raw `unknown` without narrowing** — `unknown` is allowed ONLY when immediately followed by a type guard or discriminated union check within the same function scope.
   BANNED: passing `unknown` through to other functions or returning it unnarrowed.

   ```typescript
   // ✅ Correct — narrow immediately
   function handleResponse(data: unknown): User {
     if (!isUser(data)) throw new TypeError('Invalid user data')
     return data
   }

   // ❌ BANNED — unknown passed through
   function getData(): unknown { return fetch(...) }
   ```

4. **Strict TypeScript config** — all generated `tsconfig.json` must include:
   ```json
   {
     "strict": true,
     "noUncheckedIndexedAccess": true,
     "exactOptionalPropertyTypes": true
   }
   ```
   BANNED: `@ts-ignore`. Use `@ts-expect-error` with a comment explaining the upstream bug only.

5. **Discriminated unions over boolean flags** — model state as discriminated unions.
   BANNED: `isLoading` + `isError` + `data` as separate boolean/nullable fields.

   ```typescript
   // ✅ Correct — invalid states are unrepresentable
   type State<T> =
     | { status: 'idle' }
     | { status: 'loading' }
     | { status: 'error'; error: Error }
     | { status: 'success'; data: T }

   // ❌ BANNED — allows impossible state { isLoading: true, isError: true }
   type State = { isLoading: boolean; isError: boolean; data?: User }
   ```

6. **`interface` for objects, `type` for unions/intersections/primitives** — use `interface` when defining object shapes (extendable). Use `type` for unions, intersections, mapped types, and primitives.

   ```typescript
   // ✅ Correct
   interface UserProps { name: string; age: number }
   type Status = 'idle' | 'loading' | 'error' | 'success'
   type Result<T> = Success<T> | Failure
   ```

7. **No `enum`** — use `as const` objects instead (tree-shakeable, no runtime overhead).

   ```typescript
   // ✅ Correct
   const ROLE = { ADMIN: 'admin', USER: 'user', GUEST: 'guest' } as const
   type Role = (typeof ROLE)[keyof typeof ROLE]

   // ❌ BANNED
   enum Role { Admin = 'admin', User = 'user' }
   ```

8. **Explicit return types on exported functions** — all exported functions and hooks must have explicit return type annotations. Internal functions may use inference.

9. **Readonly by default** — use `readonly` for props, function parameters, and arrays/objects that should not be mutated.

   ```typescript
   interface Props {
     readonly items: readonly Item[]
     readonly onSelect: (id: string) => void
   }
   ```

## Section 2: React Component Rules (always apply)

10. **Function components only** — use function declarations for components.
    BANNED: class components (except error boundaries pre-React 19).

    ```typescript
    // ✅ Correct
    export function UserProfile({ name, avatar }: UserProfileProps): JSX.Element {
      return <div>...</div>
    }

    // ❌ BANNED
    export default class UserProfile extends React.Component { ... }
    ```

11. **Component max 150 lines** — exceeding 150 lines signals extraction needed.
    Split into sub-components or custom hooks. A file may contain small related components if total stays under 200 lines.

12. **Function max 30 lines** — any non-component function (utils, helpers, hook logic) must be under 30 lines. Decompose if longer.

13. **Early return / guard clauses** — handle edge cases at the top with early returns. Keep happy path unindented.
    BANNED: deeply nested if/else chains (max 2 levels of nesting).
    BANNED: nested ternaries (max 1 level).

    ```typescript
    // ✅ Correct — guard clauses
    function UserCard({ user }: Props): JSX.Element {
      if (!user) return <EmptyState />
      if (user.banned) return <BannedNotice />

      return <div>{user.name}</div>
    }

    // ❌ BANNED — nested conditions
    function UserCard({ user }: Props): JSX.Element {
      return (
        <div>
          {user ? (user.banned ? <BannedNotice /> : <div>{user.name}</div>) : <EmptyState />}
        </div>
      )
    }
    ```

14. **Max 5 props before refactoring** — more than 5 props = refactor signal.
    Use: composition pattern, compound components, context, or custom hook.
    EXCEPTION: leaf UI components with primitives-only props (e.g., `<Button variant size disabled onClick children>`).

15. **Named exports only** — BANNED: `export default`.
    EXCEPTION: Next.js pages/layouts/routes that require default exports.

    ```typescript
    // ✅ Correct
    export function UserProfile({ ... }: Props) { ... }

    // ❌ BANNED
    export default function UserProfile({ ... }: Props) { ... }
    ```

16. **Co-location principle** — component, types, tests, styles live in the same directory:
    ```
    Button/
    ├── Button.tsx
    ├── Button.types.ts
    ├── Button.test.tsx
    ├── Button.module.css
    └── index.ts            # barrel export
    ```

17. **Composition over configuration** — prefer `children` and compound components over render callbacks or config objects. Build small, single-purpose components.
    BANNED: components with more than 2 render callback props.

## Section 3: Project Structure (Feature-Based Architecture)

18. **Feature-based folder structure** — big tech standard (Meta, Airbnb, Uber, Vercel):

    ```
    src/
    ├── components/           # Shared/reusable UI components
    │   └── [ComponentName]/  # PascalCase folder per component (see rule 16)
    ├── features/             # Domain features (auth, dashboard, checkout)
    │   └── [feature-name]/   # kebab-case folder per feature
    │       ├── components/   # Feature-specific components
    │       ├── hooks/        # Feature-specific hooks
    │       ├── utils/        # Feature-specific helpers
    │       ├── constants.ts
    │       ├── types.ts
    │       └── index.ts      # Public API — only export what other features need
    ├── hooks/                # Shared custom hooks
    ├── utils/                # Shared pure utility/helper functions
    │   └── helpers/          # Sub-categorized helpers (date, format, validation)
    ├── constants/            # App-wide constants
    ├── types/                # Shared TypeScript types & interfaces
    ├── services/             # API layer (fetch wrappers, endpoint definitions)
    ├── stores/               # Global state management (Zustand/Jotai stores)
    ├── lib/                  # Third-party wrappers & adapters
    └── app/                  # Routes/pages (Next.js App Router or React Router)
    ```

    **Split & Breakdown Rules:**
    - Component > 150 lines → split into sub-components
    - Function > 30 lines → extract to utils/helpers
    - 3+ components share logic → extract custom hook to `hooks/`
    - 3+ files share a constant → move to `constants/`
    - Types used across 2+ features → move to `types/`
    - Feature-specific code stays co-located; elevate to shared ONLY when reused
    - Features NEVER import from other features directly — shared code goes in top-level dirs

## Section 4: Import & Export Rules

19. **Import order (ESLint-standard, enforced)** — groups separated by blank lines:

    ```typescript
    // 1. React core
    import { useState, useCallback } from 'react'

    // 2. Third-party libraries
    import { z } from 'zod'
    import { clsx } from 'clsx'

    // 3. Internal aliases (@/)
    import { Button } from '@/components/Button'
    import { useAuth } from '@/hooks/useAuth'

    // 4. Relative imports (parent → sibling → child)
    import { formatDate } from '../utils/date'
    import { validate } from './helpers'

    // 5. Type-only imports (always last before styles)
    import type { User, UserRole } from '@/types'

    // 6. Styles
    import styles from './Component.module.css'
    ```

20. **`import type` for type-only imports** — enforced. Ensures zero runtime cost and clear intent.

    ```typescript
    // ✅ Correct
    import type { User } from '@/types'

    // ❌ BANNED — imports type as value
    import { User } from '@/types'  // when User is only used as a type
    ```

21. **Barrel exports** — every directory with 2+ public files gets an `index.ts` re-exporting the public API. Internal implementation details are NOT re-exported.
    BANNED: deep imports bypassing `index.ts` from outside the directory.

    ```typescript
    // ✅ Correct
    import { Button } from '@/components/Button'

    // ❌ BANNED — bypasses barrel
    import { Button } from '@/components/Button/Button'
    ```

## Section 5: State & Data Flow Rules

22. **No props drilling beyond 2 levels** — if a prop passes through 2+ intermediate components that don't use it, refactor to:
    - Context + custom hook (cross-cutting concerns)
    - Composition with `children`/slots (layout concerns)
    - State management store (global state)
    BANNED: passing callbacks through 3+ component levels.

23. **Custom hooks for reusable stateful logic** — any stateful logic used in 2+ components must be a custom hook.
    - Naming: `use[Domain][Action]` — `useAuthLogin`, `useCartTotal`, `useDebounce`
    - Return a well-typed object for 3+ return values (not a tuple)
    - Single responsibility — one hook = one concern

24. **Server state vs. client state** — use a server-state library (TanStack Query, SWR, or server actions) for API data.
    BANNED: `useState` + `useEffect` for data fetching.
    Client-only state (UI toggles, form drafts) stays in local `useState` or lightweight stores.

    ```typescript
    // ✅ Correct — TanStack Query
    const { data, isPending, error } = useQuery({
      queryKey: ['users', userId],
      queryFn: () => fetchUser(userId),
    })

    // ❌ BANNED — manual fetch in useEffect
    const [data, setData] = useState(null)
    useEffect(() => { fetchUser(userId).then(setData) }, [userId])
    ```

25. **Derive, do not store** — if a value can be computed from existing state/props, compute it inline.
    BANNED: storing derived values in `useState` and syncing with `useEffect`.
    Use `useMemo` only when computation is measurably expensive (>1ms).

    ```typescript
    // ✅ Correct — derived inline
    const activeCount = items.filter((item) => item.active).length

    // ❌ BANNED — stored derived state
    const [activeCount, setActiveCount] = useState(0)
    useEffect(() => { setActiveCount(items.filter(i => i.active).length) }, [items])
    ```

26. **State machines for complex flows** — multi-step flows (wizards, checkout, auth) use discriminated union state or XState.
    BANNED: multiple boolean flags for multi-step state (`isStep1`, `isStep2`, `hasSubmitted`).

## Section 6: Performance Rules

27. **No premature memoization** — BANNED: wrapping every component in `React.memo`, every function in `useCallback`, every value in `useMemo` "just in case".
    Memoize ONLY when:
    - Profiling confirms expensive re-renders
    - Reference-stable value is required for a dependency array
    - Expensive computation (>1ms measured)

28. **React Compiler first (React 19+)** — when targeting React 19+ with the React Compiler enabled, remove manual `useMemo`/`useCallback`. The compiler handles memoization automatically. Note in comments when relevant.

29. **Lazy loading** — use `React.lazy()` + `Suspense` for:
    - Route-level code splitting (always)
    - Heavy third-party components (charts, editors, maps, rich text)
    - Components not visible on initial viewport

30. **Virtualize long lists** — lists with 100+ items must use virtualization (`@tanstack/virtual`, `react-window`, or similar). BANNED: rendering 100+ DOM nodes in a flat list.

31. **Stable keys for dynamic lists** — BANNED: array index as `key` for lists that reorder, filter, or update. Use unique, stable IDs.

32. **Debounce expensive input handlers** — search inputs, resize handlers, and scroll handlers must be debounced (150-300ms). Use `useDeferredValue` for React 18+ or a custom `useDebounce` hook.

## Section 7: Modern React 19+ Patterns

> Apply these rules when targeting React 19+. Mark clearly in comments when a pattern is React 19-specific.

33. **Server Components by default** — in Next.js App Router or RSC-enabled frameworks, components are Server Components by default. Add `'use client'` ONLY when the component uses: browser APIs, event handlers, hooks (`useState`, `useEffect`, etc.), or Context consumers.

34. **`use()` hook for async data** — use the `use()` hook to unwrap promises and read context during render. Replaces `useEffect` + `useState` fetch patterns.

35. **Actions and `useActionState`** — for form submissions, use the `action` attribute on `<form>` and `useActionState` for pending/error/success state. Replaces manual `onSubmit` + `useState` boilerplate.

36. **`useOptimistic` for instant UI** — apply optimistic updates via `useOptimistic` for mutations the server will likely accept (likes, toggles, simple edits).

37. **`ref` as prop** — in React 19+, pass `ref` as a regular prop.
    BANNED: `React.forwardRef` wrapper (deprecated in React 19).

## Section 8: Testing Standards

38. **Test behavior, not implementation** — assert what the user sees and does.
    BANNED: testing internal state, hook internals, or snapshot-only tests as primary coverage.

39. **Testing Library conventions** — use `@testing-library/react`.
    Query priority: `getByRole` > `getByLabelText` > `getByText` > `getByTestId`.
    BANNED: `container.querySelector` in tests.

40. **Co-located test files** — `Component.test.tsx` lives next to `Component.tsx`.
    Custom hooks with non-trivial logic get dedicated tests via `renderHook`.

## Section 9: Naming Conventions

41. **Naming rules:**

    | Element | Convention | Example |
    |---------|-----------|---------|
    | Components | `PascalCase` | `UserProfile.tsx` |
    | Hooks | `camelCase` + `use` prefix | `useAuth.ts` |
    | Utils/helpers | `camelCase` | `formatDate.ts` |
    | Constants | `UPPER_SNAKE_CASE` | `const MAX_RETRY_COUNT = 3` |
    | Types/Interfaces | `PascalCase` | `type UserProfile = { ... }` |
    | Files | match primary export name | `UserProfile.tsx` exports `UserProfile` |
    | Feature folders | `kebab-case` | `user-profile/` |
    | Component folders | `PascalCase` | `UserProfile/` |
    | Boolean vars/props | `is/has/should/can` prefix | `isLoading`, `hasError` |
    | Event handlers | `handle` prefix (internal), `on` prefix (props) | `onClick={handleClick}` |
    | Generic type params | Descriptive or single uppercase | `<TData>`, `<TError>` |

## Section 10: BANNED Patterns Checklist

Before finalizing ANY React/TypeScript code, verify NONE of these are present:

| # | BANNED Pattern | Use Instead |
|---|---------------|-------------|
| 1 | `var` keyword | `const` |
| 2 | `let` (when `const` works) | `const` + functional patterns |
| 3 | `any` type | Proper types, generics, narrowed `unknown` |
| 4 | `export default` | Named exports (`export function/const`) |
| 5 | `React.forwardRef` (React 19+) | `ref` as regular prop |
| 6 | Class components | Function components |
| 7 | `useEffect` for data fetching | TanStack Query, SWR, server actions, `use()` |
| 8 | `useEffect` to sync derived state | Compute inline or `useMemo` |
| 9 | Props drilling > 2 levels | Context, composition, state management |
| 10 | Boolean flags for multi-step state | Discriminated unions, state machines |
| 11 | Nested ternaries (>1 level) | Early return, `if/else`, lookup objects |
| 12 | Nesting > 2 levels deep | Guard clauses, extract functions |
| 13 | Array index as `key` (dynamic lists) | Stable unique IDs |
| 14 | `@ts-ignore` | `@ts-expect-error` with explanation comment |
| 15 | `console.log` in production code | Proper logging or remove |
| 16 | Inline styles | CSS modules, Tailwind, or styled-components |
| 17 | `enum` | `as const` objects |
| 18 | `React.FC` | Explicit props type on function |
| 19 | Manual fetch in `useEffect` + `useState` | Server-state library |
| 20 | Premature `React.memo`/`useMemo`/`useCallback` | Profile first, memoize only when proven |

## DRY & Code Reuse Principles

- **Rule of Three** — duplicate code twice is tolerable. Third time → extract to shared utility/hook/component.
- **Single source of truth** — every piece of knowledge has one authoritative location.
- **Composition over inheritance** — build complex UI from small, focused components.
- **Custom hooks as the primary reuse mechanism** — prefer hooks over HOC, render props, or utility classes.
- **Shared logic hierarchy**: inline → local helper → feature util → shared hook/util → library.
